# 멀티턴 에이전틱 RAG를 위한 세션 상태 설계

단일 턴 RAG는 상태가 없다. 멀티턴은 "직전 것만 기억"을 stateless
파이프라인 위에 얹으려는 순간 깨진다. 상태는 **처음부터** 설계하라.

## 턴 사이에 무엇을 유지할 것인가

대부분의 팀은 대화 이력(사용자 + 어시스턴트 메시지)을 유지하고 끝낸다.
충분하지 않다. 최소한 다음을 유지하라:

| 필드 | 이유 |
|---|---|
| `shown_documents` | 사용자가 실제로 본 참조 문서들. "더 보여줘"에 같은 문서를 다시 권하지 않기 위함. |
| `last_shown_documents` | 직전 턴의 순서 보존된 참조들. "2번째 문서"를 위치로 해석할 수 있게. |
| `last_filters` | 직전 턴에 적용된 부서/문서/기간 필터. 후속 턴은 이들을 정제하고 싶어한다. |
| `last_records` | 직전 검색의 상위 N개 원본 구조화 행. 재검색 없이 서브셋 필터링을 가능하게. |
| `last_distributions` | 사전 계산된 집계 카운트(상태별/팀별/품목별). 분석성 후속 질문에 재검색 없이 답할 수 있게. |
| `active_topic` | 현재 대화 포커스의 짧은 사람이 읽을 수 있는 라벨. 토픽 드리프트 방지. |
| `persistent_exclude` | 교차-턴 제외("바이오 제외"). 사용자가 명시적으로 철회할 때까지 유지. |
| `disambiguation` | 용어 → 선택된 의미. 세션 동안 고정. |

## 핵심 통찰: 요약이 아니라 원본 행을 유지하라

사용자가 2턴에 "고형제만 추려줘"라고 하면, 에이전트는 1턴의 실제 행을
필터링해야 한다. 텍스트 요약만 유지했다면 에이전트는 재검색할 수밖에
없고 — 재검색은 직전 필터 컨텍스트를 잃는다.

턴당 20개 구조화 행을 유지하는 비용은 검색 확장의 비용에 비하면 사소하다.

## 핸드오프 포맷이 중요하다

레코드 유지는 1단계다. 2단계는 다음 턴 에이전트에게 **작동 가능한**
형태로 노출하는 것 — 단순 주변 컨텍스트가 아니다.

나쁜 예:
```
Previously shown 10 records.
```

좋은 예:
```
[PRIOR RECORDS — FILTERABLE]
Source query: 원료 이물 일탈
Records: 10

  [1] PR-XXXXXX | Major | 생산1팀 | 종결 | 2025-03-31
      A정 10mg 원료 이물 발견 건
  [2] PR-YYYYYY | Major | 생산2팀 | 종결 | 2025-02-14
      B정 5mg 2차 코팅액 제조 중 이물 ...
  ...
```

차이점: 두 번째 형식은 에이전트에게 "이것은 행이다, 필터링할 수 있다"를
말해준다. 첫 번째 형식은 LLM이 다시 파싱해야 하는 텍스트일 뿐이다.

## 세션 TTL과 정리

- 24시간 TTL은 대부분 채팅 UX에 합리적이다.
- 30분마다 정리. 매 요청마다 하지 말라.
- 컨테이너 재시작 생존: SQLite에 쓰면서 hot 인메모리 캐시 유지.
- 무한 증가 주의: `shown_documents`는 최근 10개, `topic_history`는 최근
  5개, `rejected_documents`는 최근 5개로 캡.

## 유지하지 **말아야** 할 것

- 도구의 전체 출력 — 너무 크고 대부분 중복이다.
- LLM 원본 출력 — 재도출 가능하고, 재사용하고 싶지 않은 추론을 유출할 수 있다.
- 사용자가 명시적으로 저장을 동의하지 않은 민감값.

## 후속 턴(followup) 감지

"이 턴이 followup인가?"는 보이는 것보다 어렵다. "직전 어시스턴트 메시지가
있었나?"로 판정하지 말라 — 그건 첫 턴 이후 언제나 참이다. 더 나은
신호들:

- 직전 턴과의 토픽 유사도(재작성된 쿼리의 임베딩 유사도).
- 명시적 참조("이중", "아까", "방금", "위에서").
- 서브셋 필터 키워드("만", "X만", "그럼 X는").

의심이 들 때는 **"예, followup이다"** 쪽으로 편향하라 — 직전 컨텍스트를
잘못 노출해도 비용은 낮지만(에이전트가 무시하면 된다), followup을 놓치면
비용이 크다(에이전트가 넓게 재검색하고 컨텍스트를 잃는다).
