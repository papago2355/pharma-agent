# 예시 토큰 누출 — 잘 쓴 프롬프트가 너무 문자 그대로 해석될 때

> 규칙은 깔끔하다. 예시도 정확하다. 그런데 사용자가 그 예시와 같은
> 단어를 쓰는 모든 쿼리에서 모델이 똑같이 어긋난다.

규칙도 옳고 예시도 옳다. 그런데 프로덕션 트레이스를 보면, 모델이
**전제 조건이 명백히 충족되지 않은** 쿼리에서도 그 규칙을 적용한다.
공통점은 하나뿐이다. **사용자 쿼리에 예시의 명사가 들어 있다.**

이 현상이 **예시 토큰 누출(example-token leakage)**이다. 모델이 few-shot
예시 안의 구체적인 명사를 *플레이스홀더*가 아니라 *트리거 토큰*으로
취급하는 것.

## 실제 사고 사례

프로덕션 규칙 (한국 제약 RAG, 2026-04):

> **Rule 13 (EXCLUSION)**: 사용자가 **제외 / 말고 / 빼고 / 없이**라고
> 말하면, 제외 대상 단어를 `exclude_terms` 파라미터에 전달한다.
> *예시*: **`"바이오 부서를 제외한 멸균 문서"`** → `exclude_terms: ["바이오"]`.

전제 조건은 명확하다 — 쿼리에 제외 동사가 있어야 한다. 예시도 모호하지
않다. 교과서 같은 규칙이다.

시스템을 깬 사용자 쿼리:

```
"[바이오] 협력업체 관리 [바이오] 위수탁 시험 관리 방법서(바이오QC)
 [바이오] 품질협약서 관리 요약해줘"
```

`제외 / 말고 / 빼고 / 없이` 어디에도 없다. `[바이오]` 대괄호는 세 개의
SOP 파일명의 일부일 뿐 — 사용자는 그 세 문서를 **찾아서** 요약해 달라고
요청하고 있다.

에이전트가 보낸 액션:

```json
{ "action": "search",
  "params": { "query": "협력업체 관리 위수탁 시험 ...",
              "exclude_terms": ["바이오"] } }
```

이후 후처리 필터가 검색 결과 10개 중 5개를 떨어뜨렸다 — 사용자가
명시적으로 이름을 댄 `[바이오] …` PDF 세 개가 모두 그 5개에 포함되어
있었다. 에이전트는 같은 주제의 비-바이오 문서들을 요약하면서 사용자가
요청한 바이오 문서는 "찾을 수 없다"고 답했다. 같은 세션의 후속 질문에서는
정상 동작했고, 운영 엔지니어는 "모델이 가끔 흔들린다"는 유령을 쫓게
됐다.

같은 프롬프트로 새 프로세스에서 재현했을 때 에이전트가 출력한 `thought`
스트림이 메커니즘을 정확히 보여준다:

> "사용자가 '바이오' 관련 ... 내용을 요청했습니다. 하지만 **'바이오'라는
> 키워드가 포함되어 있으므로**, 우선 `exclude_terms`에 '바이오'를 넣어
> 검색하여 ..."

모델은 규칙의 전제 조건(사용자가 제외 동사를 썼는가)이 아니라 **쿼리에
"바이오" 토큰이 있는지**를 기준으로 적용 여부를 저울질한다.
온도 0의 fresh 프로세스에서도 의미 있는 비율로 잘못된 분기를 골랐다.

## 왜 이런 일이 생기나

LLM은 "이것이 전제 조건이다"와 "이것이 전제 조건의 한 구체적 사례다"를
안정적으로 분리하지 못한다. Few-shot 예시는 둘 다 담고 있다. 모델이
학습한 사람 글쓰기에서 예시는 *그것이 설명하는 사례와 어휘를 공유함으로써*
설명을 한다. 추론 시점에서 모델은 예시의 토큰을 유사도 앵커로 사용하고,
규칙이 명시한 전제 조건은 여러 신호 중 하나로 약화된다.

증폭 조건:

- 예시 토큰이 **광역 코퍼스에서는 희귀하지만 도메인 안에서는 흔할 때**
  (예: 제약 SOP의 `바이오`). 모델이 예시 사용과 실제 사용을 구분하기
  어렵다.
- 규칙에 **예시가 하나뿐**일 때. 비교/삼각측량할 다른 사례가 없다.
- 규칙 본문이 짧을 때. 예시의 끌어당김을 상쇄할 컨텍스트가 부족하다.
- 예시에서의 토큰 역할이 사용자가 그 토큰을 쓰는 방식과 겹칠 때
  (여기서는: 규칙 예시의 `바이오`도 부서명, 사용자 쿼리의 대괄호
  `[바이오]`도 부서/카테고리 표지).

## 고침은 코드가 아니라 프롬프트에서

먹히지 않는 유혹들:

- `if "[" in query: skip_exclude` — 하드코딩 라우팅. 다음 표현 변종에서
  깨지고, 의미적 결정은 LLM에게 맡긴다는 원칙을 무너뜨린다.
- `exclude_terms` 사용 여부를 분류기가 게이트하게 만들기. 버그를 제거하지
  않고 라우터를 그 버그 둘레에 다시 짓는 것이다.
- 온도를 0으로 낮추기. 우리가 했다. 이 정확한 쿼리에 대해 프로덕션
  트레이스에서도 버그가 발생했다 — 온도는 누출을 줄이지만 없애지는
  않는다.

같이 적용해서 효과를 본 세 가지 프롬프트 변경:

1. **전제 조건을 부드러운 힌트가 아니라 단단한 게이트로 만들기.**
   ```
   exclude_terms는 사용자 쿼리에 다음 중 하나가 LITERALLY 포함될 때에만
   설정한다: 제외 / 말고 / 빼고 / 없이.
   ```
   "사용자가 제외와 비슷한 말을 할 때"라고 쓰지 말고 트리거 토큰을
   열거한다. 모델은 패러프레이즈된 문장보다 화이트리스트를 훨씬
   안정적으로 따른다.

2. **예시의 명사를 충돌이 큰 토큰에서 빼기.**
   예시를 `바이오 부서를 제외한 멸균 문서`에서 `주사제 부서를 제외한
   멸균 문서`로 바꿨다. `주사제`도 이 도메인의 부서로 충분히 그럴듯하지만
   대괄호 파일 접두사로 등장할 가능성은 훨씬 낮다 — 거짓 트리거 표면을
   제거한다.

3. **반-패턴을 명시적으로 부른다.**
   ```
   대괄호 접두사(`[바이오]`, `[QC]`, `[VAL]`, `[페니]`, `[OSD]` …)는
   POSITIVE 선택자이지 제외 표지가 아니다. `"[바이오] X 관리"` 같은
   쿼리는 X에 대한 `[바이오]`-접두사 문서를 요청하는 것이다.
   ```
   모델이 만들고 있는 정확한 거짓 양성 케이스를 직접 호명해야 할 때가
   많다 — 암시적 가이드("과적용하지 마라")만으로는 부족하다.

## 그리고: 에이전트에게 양극(positive) 레버를 주기

더 깊은 구조적 원인 — 에이전트의 `search` 액션은 `exclude_terms`만
있고 양극 어휘 필터가 없었다. 망치만 있으면 모델은 망치를 휘두른다.
`title_contains`(파일명 Milvus 전치 필터, 항목 간 AND)를 추가하니 모델이
*"사용자가 `[바이오]`-접두사 문서를 원한다"*를 표현할 수 있게 됐다 —
그게 없었을 때는 스키마 안의 유일한 카테고리 필터 레버가 `exclude_terms`
하나였다.

```yaml
# 이전: 필터가 하나, 이 케이스에는 잘못된 극성
search:
  params:
    query:         { type: string, required: true }
    exclude_terms: { type: array,  items: string }

# 이후: 대칭된 극성. 에이전트가 양극 선택을 표현할 수 있다
search:
  params:
    query:                  { type: string, required: true }
    exclude_terms:          { type: array,  items: string }
    title_contains:         { type: array,  items: string }   # AND
    title_contains_any:     { type: array,  items: string }   # OR
```

두 수정(프롬프트 전제-조건 게이트 + 새 양극 레버) 이후, 원래 실패했던
쿼리를 연속으로 세 번 재현해 봤더니 모두 `title_contains: ["바이오"]`
호출로 떨어졌고, 사용자가 이름 댄 세 PDF가 점수 0.99 / 0.97 / 0.95로
최상위에 떴으며 `exclude_terms`는 비어 있었다.

## 일반화된 교훈

worked example가 들어간 규칙이 *예시와 어휘가 겹치는 쿼리에서 일관되게*
어긋난다면, 예시 토큰이 트리거 조건으로 누출되고 있는 것이다. 할 수 있는
세 가지:

1. **Few-shot 예시를 감사하라**. 예시의 명사가 무관한 이유로 실제
   사용자 쿼리에 등장할 수 있는가? 그렇다면 명사를 바꿔라.
2. **모호한 전제 조건을 토큰 리터럴 리스트로 바꿔라.** "사용자가 제외를
   언급할 때" → "쿼리에 제외 / 말고 / 빼고 / 없이가 포함될 때." 짧은
   규칙 + 긴 enum이 긴 규칙 + 짧은 enum보다 낫다.
3. **도구 스키마에 모든 카테고리 레버의 양극이 다 있는지 확인하라.**
   스키마가 음극 선택만 표현하게 한다면, 모델은 관련된 모든 쿼리를
   음극-선택 문제로 변환할 것이다.

## 출시 전에 테스트하라

이 버그를 미끼로 거는 행동 테스트를 함께 만든다:

```yaml
# benchmark scenario
turns:
  - user: "[바이오] 협력업체 관리 [바이오] 품질협약서 관리 요약해줘"
    expect:
      tool_called: search
      tool_params_not_contain: { exclude_terms: ["바이오"] }
      answer_contains: ["[바이오] 품질협약서 관리"]
```

결정론적 한 턴이면 충분하다 — 멀티턴 상태 버그가 아니라 리터럴
전제-조건 게이트의 문제다. 게이트 프롬프트 + 중립 예시가 들어가면
이 테스트는 안정적으로 통과한다. 없으면 같은 모델이 같은 온도에서도
의미 있는 비율로 이 턴에서 실패한다.

실행 가능한 버전은
`skills/korean-multiturn-rag/benchmarks/behavioral/scenarios/s12_example_token_leakage.yaml`
참조.
