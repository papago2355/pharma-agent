# 에이전틱 RAG의 한국어 NLP 함정들

대부분의 에이전틱 RAG 가이드는 영어로, 영어를 위해 쓰였다. 사용자가
한국어로 입력하면 영어권 데모에서는 절대 보이지 않는 실패 모드를 하나씩
만난다. 각 항목은 내가 며칠씩 날려먹은 경험이다.

## 1. 조사가 필터를 잡아먹는다

한국어는 **조사**를 명사에 붙인다. "고형제만 추려줘"는 `고형제 + 만`이다
— `만`은 "only"를 뜻하는 조사다. 에이전트가 검색 레이어에
`title_contains=["고형제만"]`을 넘기면? **0건**이 반환된다. 제목에
`고형제만`이라는 문자열을 가진 문서는 세상에 없기 때문이다.

에이전트가 쿼리에서 뽑아내는 모든 문자열 값 필터는 검색에 들어가기 전에
**조사 제거(particle stripping)**를 거쳐야 한다:

| 사용자 표현 | 조사 | 실제 필터 값 |
|---|---|---|
| `고형제만` | 만 | `고형제` |
| `서울에서` | 에서 | `서울` |
| `QC팀의` | 의 | `QC팀` |
| `변경관리에 대한` | 에 대한 | `변경관리` |
| `바이오 부서를` | 를 | `바이오 부서` |

**실패 모드는 조용하다.** 검색은 0건을 반환하고, 에이전트는 "데이터가
없다"고 판단하고, 사용자는 "해당 기록이 없습니다"를 본다 — 바로
옆에 있는 데이터에 대해서 말이다.

**파이썬 전처리가 아니라 스키마를 고쳐라.** 이걸 도구 설명에 넣어라:

> "`title_contains`를 추출할 때 한국어 조사(만/의/에서/에/을/를/이/가/은/는)를
> 제거하라. 조사는 값의 일부가 아니다."

LLM이 하게 하라. 한국어 형태소는 너의 regex보다 LLM이 낫다.

## 2. 카테고리 이름은 데이터에 없다

제약 도메인에서 **제형** 카테고리(고형제/액제/주사제)는 사람들이 말할 때
쓰는 단어다. 하지만 문서는 **제품명 토큰**을 쓴다 — `정`, `캡슐`, `환`,
`과립`, `주`, `시럽`, `점안` — 실제 제품명(`A정`, `B캡슐`,
`C주`)에 이게 들어가기 때문이다.

결과: `title_contains=["고형제"]`는 수천 건의 고형제 레코드가 있는
데이터베이스에서 0건을 반환한다. 카테고리 라벨은 제목에 없다.

**올바른 핸드오프 패턴:**

1. 첫 시도: 사용자의 표현을 존중 — `title_contains=["고형제"]` 시도.
2. 0건 반환되면: 카테고리-토큰 매핑을 위한 위키 페이지를 참조
   (고형제 → 정/캡슐/환/과립/정제).
3. `title_contains_any=["정","캡슐","환","과립"]`로 재시도.
4. 최종 답변에서 카테고리를 특정 제형 토큰으로 확장했음을 **사용자에게
   알려라** — 대체를 숨기지 말라.

이것은 위키 증강 검색 패턴이다. 매핑은 에이전트가 inference 시점에
읽는 마크다운 페이지에 살아야 한다 — 하드코딩이 아니다.

## 3. 머뭇거림(hedge) 구절 회귀

한국어 LLM은 — 특히 QMS/SOP 콘텐츠에서 — 이런 구절을 사랑한다:

- "직접적인 내용은 확인되지 않습니다"
- "관련된 명시적 언급은 없으나..."
- "해당 SOP만으로는 파악이 어렵습니다"
- "추가 확인이 필요합니다"

때로는 맞다. 하지만 **종종 모델이 반사적으로 적용하는 회피**다 — 검색된
청크에 특정 조항 번호까지 포함된 정확한 답이 있는데도 말이다. 사용자는
"직접적인 내용은 없습니다"를 읽고 시스템이 작동하지 않는다고 결론짓는다.

**수정 방향 (코드가 아니라 프롬프트):**

- 검색된 청크가 실제로 비어있지 않은 한 `"직접적인 내용은 없"` 구절을
  생성 프롬프트에서 **명시적으로 금지**하라.
- 가이드를 다시 써라: "청크에 부분 정보가 있다면, 부분 정보를 인용하고
  섹션을 언급하라 — '직접적인 내용은 없다'고 회피하며 물러서지 말라."
- 청크가 질문의 절반만 다루어도 5.1.4항을 정확히 인용하는 few-shot
  예시를 추가하라.

## 4. 토크나이저 분산이 하이브리드 검색을 깨뜨린다

하이브리드 검색(dense + sparse)은 희소 토크나이저가 인덱싱 시점과 쿼리
시점에 동일한 토큰을 만든다고 가정한다. 한국어는 이걸 깬다:

- BGE-M3 빌드에 따라 `정제`가 `정제` (1토큰) vs. `정`+`제` (2토큰)으로
  다르게 쪼개질 수 있다.
- 형태소 분석기(Mecab, Kiwi, Khaiii)들은 어디서 자를지 서로 의견이 다르다.
- Unicode 정규화(NFC vs NFD)가 한글 음절의 바이트 시퀀스를 바꾼다 —
  시각적으로 같아 보이는 문자열이 서로 다른 바이트일 수 있다.

**증상:** 지난 주에는 완벽히 랭킹되던 코퍼스가, 같은 쿼리에 대해 15위로
떨어진다. 코드는 한 줄도 안 바뀌었다. 토크나이저 패치가 조용히 배포된
것이다.

**방어책:**

- 한글을 **인덱싱 시점과 쿼리 시점 둘 다**에서 정규화(`unicodedata.normalize("NFC", s)`).
  한 형태를 고르고 고정하라.
- 희소 임베딩 모델 버전을 **핀으로 고정**하라. `:latest`를 쓰지 말라.
- 임베딩을 재빌드할 때마다 작은 golden-query 스위트를 돌리고 이전
  baseline과 top-10 diff를 찍어라. 임계값을 넘으면 재빌드를 중단하라.

## 5. 띄어쓰기는 때로 의미를 바꾼다

"고 형제" vs "고형제" — 하나는 "큰 형제"(말이 안 되는 소리)이고 다른
하나는 "고형 제형"이다. 오타 하나가 전부를 바꾼다. Milvus exact-match
필터는 사용자가 띄어쓰기를 잘못 넣는 순간 제대로 된 문서를 기쁘게
놓쳐버린다.

덜 분명한 예: `바이오QC` vs `바이오 QC` vs `바이오-QC` — 다 흔하게 쓰이고,
문자열 매칭으로는 전부 다르다.

**해결책:**

- `title_contains`를 위해 알려진 부서/팀 이름의 **띄어쓰기 있는/없는
  변형**을 둘 다 인덱싱하라.
- 쿼리 시점에 사용자 문자열을 synonym 테이블(위키!)에 대조해보고
  포기하라. "바이오QC"가 0건이고 "바이오 QC"가 synonym에 있다면 재시도하라.
- 절대 regex로 이 로직을 만들지 말라. synonym 테이블을 LLM이 읽는
  위키 페이지에 두라.

## 6. 동음이의어가 stem을 무너뜨린다

- `정` → 錠 (정제), 正 (바를 정), 情 (느낌 정), 이름 접미사 등.
- `주` → 주사제, 週 (주간), 株 (주식), 주님.
- `일탈` → QMS 용어로는 "편차/일탈", 일반 한국어로는 "탈선/벗어남".

문맥이 모호성을 해결한다 — 그러므로 **코드로 모호성을 풀려 들지 말라**.
시스템 프롬프트에 도메인 컨텍스트를 주면 LLM이 월등히 잘 한다.

## 7. 영어 프롬프트 → 한국어 출력이 보통 더 낫다

직관과 어긋나지만 사실: 많은 한국어 LLM(Qwen, Gemma 계열)은 한국어
지시보다 **영어 지시를 더 안정적으로 따른다**. 같은 모델, 같은 태스크,
같은 입력 — 영어 시스템 프롬프트 + `output in Korean`이 더 깔끔한
구조를 만든다.

이유: 모델의 instruction-tuning 데이터에는 한국어 지시 패턴보다 영어
지시 패턴이 훨씬 많다. "bulleted list를 따르라"를 한국어보다 영어로
훨씬 더 철저하게 학습한 것이다.

**우리가 쓰는 패턴:**

- 시스템 프롬프트: 영어 지시 + 규칙 + few-shot 예시.
- 끝에 명시적 지시: `"Always write the final answer in natural Korean
  (한국어로 답변)."`
- 도구 observation: 가능한 곳은 영어로 유지.

한국어 사용자는 영어 부분을 절대 보지 못한다. 그들은 — 한국어-only
프롬프트가 만들어낼 수 있는 것보다 — 규칙을 더 안정적으로 따른 깔끔한
한국어 출력만 본다.

## 8. 브래킷 누출 회귀 (모델별)

Gemma-4는 한국어 콘텐츠에서 프롬프트에 쓴 브래킷 주석을 사용자 출력에
**그대로 복사해 넣는다**:

- 생성 프롬프트에 `[NOTE: cite the page number]`라고 쓴다.
- Gemma가 내놓는다: `"... 5.1.4항 [NOTE: cite the page number]에 따르면..."`.

브래킷 내용이 렌더된 답변에 그대로 박힌다. 사용자가 내부 주석을 보게
된다.

**방어책:**

- Gemma에서는 프롬프트 안에 `[NOTE:...]`, `[HINT:...]`, `[IMPORTANT:...]`를
  쓰지 말라. 인라인 명령형을 써라 ("Always cite the page number.").
- 최종 답변에서 `\[NOTE:[^\]]+\]` 패턴을 제거하는 후처리 regex를 추가하라
  (regex의 몇 안 되는 정당한 용도 중 하나다 — 의미 라우팅이 아니라
  데이터 위생이다).
- 모델을 바꿀 때마다 smoke test를 돌려라: 출력에 `[` 가 있는지 본다.

## 출시 전 빠른 체크리스트

- [ ] 에이전트가 설정하는 모든 문자열 필터가 파이썬이 아니라 **프롬프트
      레벨에서 조사 제거**를 거치는가?
- [ ] 모든 한국어 카테고리 라벨에 **위키 기반 토큰 폴백**이 있는가?
- [ ] 생성 프롬프트가 너의 LLM이 남용하는 **머뭇거림 구절을 명시적으로
      금지**하고 있는가?
- [ ] 한글이 인덱싱과 쿼리 모두에서 **정규화(NFC)**되는가?
- [ ] 희소 임베딩 모델 **버전이 핀 고정**되어 있는가?
- [ ] 임베딩 재빌드 후 **golden-query diff**를 돌리는가?
- [ ] 부서/팀 동의어가 `if/elif`가 아니라 **위키 페이지**에 있는가?
- [ ] Gemma를 쓴다면: 프롬프트에 `[NOTE:...]` 없고, 출력에 브래킷 제거
      regex가 있는가?
