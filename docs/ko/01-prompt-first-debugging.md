# 프롬프트 우선 디버깅 (Prompt-first debugging)

> 에이전트가 이상하게 동작할 때, 첫 번째 가설은 **"프롬프트나 도구 스키마에
> 정보가 빠져 있다"**여야 한다. **"코드에 버그가 있다"**가 아니다.

대부분의 팀은 코드부터 손댄다. 에이전트가 엉뚱한 도구를 골랐다 →
`if query.contains("X")` 분기를 추가한다. 검색이 너무 많은 행을 반환했다
→ 후처리 필터를 붙인다. 에이전트가 구체적인 답을 피한다 → 폴백 템플릿을
추가한다.

이런 패치 하나하나가 LLM의 동작을 **떠받치는 비계**가 된다. 몇 달이 지나면
"에이전트"는 더 이상 에이전틱하지 않다 — LLM이 껍데기로 붙은 하드코딩
상태 기계일 뿐이다. 다음 모델 업그레이드가 도착하면 그 비계 전부가
잡음이 된다.

## 조사 순서

1. **도구 파라미터가 누락된 것이 아닌가?**
   기존 액션에 새 파라미터를 추가하면 사용자의 요구를 에이전트가 표현할 수
   있는가? `title_contains` 필터 하나가 `if "고형제" in query:` 분기보다
   거의 항상 낫다.

2. **시스템/레시피 프롬프트에 지시문이 빠진 것은 아닌가?**
   에이전트가 이 케이스의 존재 자체를 아는가? 문장 하나 + few-shot 예시
   하나가, 검증 코드 30줄보다 효과가 크다.

3. **에이전트가 보는 observation에 필요한 정보가 빠진 것은 아닌가?**
   카운트, 분포, 스키마 힌트, 직전 턴의 컨텍스트를 observation summary에
   직접 노출하라 — 에이전트가 읽을 수 없는 숨겨진 상태로 두지 말라.

4. **위 1–3을 모두 소진한 후에만**: 코드를 쓴다. 그때도 기존 함수 안에
   고정된 if/else를 박는 것보다 **새 작은 파라미터**를 추가하는 편을 택하라.

## "정말로 LLM의 한계인가?" 테스트

코드 패치를 쓰기 전에 아래 세 가지 질문에 모두 답하라:

1. **재현성**: 같은 모델에 더 나은 프롬프트나 파라미터 하나를 추가하면
   이 케이스를 맞힐 수 있는가? 답이 "예"라면, 모델 한계가 아니라 프롬프트
   공백이다.
2. **일반성**: 내가 제안한 코드 패치가 케이스 하나만 해결하는가, 아니면
   실제 입력군 전체를 커버하는가? 단일 케이스 패치는 거의 항상 프롬프트
   이슈를 코드로 위장한 것이다.
3. **회귀 안정성**: 이 패치가 모델 업그레이드나 레시피 리팩토링을 버티는가?
   하드코딩된 라우팅 테이블과 고정 if-else는 에이전트 동작이 바뀌는
   순간 썩는다.

세 질문 모두에 "예"라고 답할 수 없다면, 수정은 `services/` 코드가 아니라
**프롬프트나 도구 스키마에** 들어가야 한다.

## "프롬프트로 돌아가라"는 증상들

| 증상 | 수정 레이어 |
|---|---|
| "에이전트가 X를 모른다" | 시스템 프롬프트 |
| "에이전트가 엉뚱한 필드를 넣는다" | 도구 스키마 설명 |
| "에이전트가 엉뚱한 도구를 고른다" | 액션 설명 / 레시피 규칙 |
| "에이전트가 카운트를 잘못 센다" | observation summary에 카운트 노출 |
| "에이전트가 직전 컨텍스트를 무시한다" | observation metadata로 전달 |
| "사용자는 X라고 했는데 에이전트는 Y를 검색했다" | few-shot 예시 추가 |

## 진짜 코드가 필요한 증상들

- 토큰 컨텍스트 오버플로우 — 실제 엔지니어링 제약.
- 프론트엔드가 의존하는 스트리밍 프로토콜 계약.
- 검증자가 잡아내긴 하는데 프롬프트로는 막을 수 없는 환각.
- 데이터 필드 자체가 스키마에 존재하지 않는 불일치.
- 무한 루프를 막기 위한 버짓 강제.

이 목록에 없다면, 아마도 첫 번째 목록에 속한다.

## "if 한 줄인데 뭐 어때"의 함정

모든 하드코딩 분기는 그 자체로는 무해해 보인다. 함정은 **누적**이다.
30개가 쌓이고 나면, 라우팅은 불투명하고 테스트는 깨지기 쉬우며 다음
모델 업그레이드는 그 30개를 하나씩 다시 증명해야 한다.

특히 비영어 언어에서 심각하다. 한국어는 형태소 결합, 조사 부착, 띄어쓰기
변이 때문에 regex/키워드 라우팅이 구조적으로 불안정하다. 추가하는 모든
`if "X" in query:`는 시한폭탄이다.

## 예외: 순수 데이터 검증

이 원칙은 **의미론적 결정**에 적용된다 — 의도 탐지, 콘텐츠 분류, 관련성
판정. 순수 데이터 검증(필드가 비어있지 않은지 확인, 알려진 와이어 포맷
파싱, 날짜 형식 검증)은 코드에서 해도 된다. 경계선은 이렇다: 문자열
체크로 **사용자 의도를 분류**하고 있다면 그건 LLM이 해야 할 일이다.
JSON 필드 존재 여부를 확인한 뒤 역참조한다면 그건 데이터 위생이다.
