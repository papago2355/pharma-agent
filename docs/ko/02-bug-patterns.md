# 운영 환경 에이전틱 RAG의 버그 패턴

각 항목은 며칠씩을 잡아먹었던 사례들이다. 증상은 종종 오해를 부르고,
실제 수정 지점은 처음 보이는 곳이 거의 아니다.

## 1. thinking 로그가 최종 답변보다 좋다

**증상**: 에이전트의 thinking 패널에는 올바른 정보가 있는데, 최종 답변은
머뭇거리거나, 더 짧거나, 틀려 있다.

**근본 원인**: 생성(generation) LLM은 별도 호출로 돌아가고, 에이전트가
본 모든 것을 보지는 못한다. 에이전트의 추론이 참조한 데이터가 생성기가
읽을 수 없는 곳(세션 상태, 직전 턴의 observation, 사용자 쿼리에 붙여
넣어진 필터 도구 컨텍스트)에 살고 있다면, 생성기는 부분적으로만 보이는
레코드와 "데이터 없음, 답변 거부" 지시를 동시에 받아 — 둘의 절충안을
내놓는다.

**수정 방향**: 생성기의 컨텍스트를 에이전트의 컨텍스트의 **상위집합**으로
만들어라. 직전 레코드를 synthetic step-0 observation으로 주입하거나,
직전 턴 컨텍스트를 생성 프롬프트에 명시적으로 전달하라.

## 2. 후속 턴이 필터링 대신 검색을 넓힌다

**증상**: 1턴은 타깃된 11개 레코드를 반환한다. 2턴("X만 추려줘")은
일반적인 5529개 레코드를 반환한다. 답변은 우연히 1턴 기억에서 맞는
행을 인용하지만, 참조 패널에는 관련 없는 결과가 뜬다.

**근본 원인**: 에이전트의 followup 템플릿이 직전 턴의 원본 레코드를
노출하지 않는다. 재검색은 직전 토픽 필터를 잃는다. 에이전트에게 "이미
가진 것에서 필터링하고 싶다"를 표현할 수단이 없다.

**수정 방향**: 세션 상태에 상위 N개 직전 레코드를 유지하라. followup의
시스템 프롬프트에 "[PRIOR RECORDS — FILTERABLE]" 블록으로 노출하라.
서브셋 요청 시 thought 안에서 필터링을 허용하도록 프롬프트 규칙을
업데이트하라.

## 3. 검증자가 교차-턴 추론을 오탐한다

**증상**: 답변은 맞는데 검증자가 "fabricated"라고 플래그한다. 검증자는
현재 턴 컨텍스트에서 찾을 수 없는 숫자를 지적한다.

**근본 원인**: 검증자는 현재 턴의 생성 컨텍스트만 받는다. 답변이 직전
턴 데이터에 대해 정당하게 추론하면, 그런 주장은 전부 검증자에게 환각으로
보인다.

**수정 방향**: 직전 턴 컨텍스트를 검증자 프롬프트에 **명시적으로 라벨링된
별도 섹션**으로 전달하라. 직전 턴 증거를 유효한 근거로 취급하도록 검증자
규칙을 업데이트하라.

## 4. 고득점 검색에도 머뭇거리는 톤

**증상**: 참조 패널에는 매우 관련 높은 문서(점수 0.8+)가 뜨는데, 답변은
"관련 정보를 찾을 수 없습니다" 혹은 "직접적인 내용은 없으나..."로
회피한다.

**근본 원인**: 보통은 검색 문제가 아니다 — 프롬프트 문제다. 생성
프롬프트의 보수적인 문구("컨텍스트에 있는 것만 인용", "절대 지어내지 말 것")를
LLM이 과잉 적용한다. 청크에 부분 정보가 있어도 → 부분 정보를 인용하는
대신 LLM이 거부한다.

**수정 방향**: "정보 없음"과 "부분 정보"를 구분하도록 생성 프롬프트를
다시 써라. 페이지 번호, 섹션 번호, 부분 인용을 명시적으로 허용하라.
회피 표현에서 "청크만 봤다"류 구절을 금지하라.

## 5. 모델 업그레이드 후 썩는 하드코딩 라우팅

**증상**: `if "keyword" in query:` 분기로 가득한 라우팅 코드. 한 모델에선
동작한다. 모델을 교체하면 조용히 회귀한다.

**근본 원인**: 모든 하드코딩 분기는 LLM의 토크나이징과 응답 방식에 대한
가정을 코드에 새긴다. 새 모델은 그 가정을 보이지 않게 깨뜨린다 — 실패하는
테스트는 없는데 특정 쿼리가 어긋난다.

**수정 방향**: 라우팅 분기를 LLM이 inference 시점에 읽는 도구 스키마와
레시피 설명으로 대체하라. 분기가 꼭 필요하다면 **LLM이 선택하는 도구
파라미터**로 만들라 — 코드 레벨 체크가 아니다.

## 6. 타입드 섹션의 step-number 불일치

**증상**: 에이전트가 레코드 0개인 step을 참조하는 타입드 섹션 `final_answer`를
만든다. 생성기는 빈 테이블을 렌더링한다.

**근본 원인**: 에이전트가 휴리스틱으로 step 번호를 골랐는데(예: "step 2가
성공적인 검색이었다") 그 step은 0개 결과의 재시도였다. 실제 레코드는 step 1에
있다.

**수정 방향**: (a) 생성기에 fallback — 참조된 step에 레코드가 0개면 모든
non-final step을 사용. (b) 에이전트가 `steps[]`를 완전히 생략할 수 있게
(기본값은 전체).

## 7. 과다 검색으로 인한 컨텍스트 오버플로우

**증상**: 검색된 청크가 특정 개수를 넘으면 답변 품질이 급락한다. 구체적인
문서가 검색됐는데도 답변은 일반적이 된다.

**근본 원인**: 생성 컨텍스트가 모델 한계에 가까워졌다. LLM이 청크를 조용히
잘라내거나 핵심 디테일에 주의를 잃는다.

**수정 방향**: 생성 프롬프트의 총 토큰 수를 로깅하라. 청크를 **관련성
점수로** 캡하라 — 개수로 캡하지 말라. 한계 근처 청크 여러 개보다 고품질
청크 몇 개를 택하라.

## 8. SSE 스트리밍이 도중에 끊긴다

**증상**: 사용자가 갑자기 끊긴 부분 답변을 본다. 로그에 오류 없다. 같은
쿼리를 재시도하면 완전한 답변이 나온다.

**근본 원인**: 다운스트림 클라이언트의 read timeout, 또는 LLM이 스트림
도중 컨텍스트 한계를 친 것. 때로는 프롬프트에 포함된 도구 결과가 모델의
입력 버짓을 초과한다.

**수정 방향**: (a) 스트리밍 시작 전 프롬프트 크기를 줄여라. (b) 서버 측
타임아웃 헤더를 추가하라. (c) SSE 스트림에 클라이언트가 기다릴 수 있는
`done` 이벤트를 계측하라 — 없으면 부분 출력을 조용히 표시하지 말고
"연결 끊김, 재시도?" 프롬프트를 띄워라.
