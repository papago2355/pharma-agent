# 제약 도메인 특화 패턴 (agentic RAG)

규제 산업의 RAG에는 일반 쿡북이 다루지 않는 규칙들이 있다. **인용은
감사(audit)를 통과해야 하고**, 일탈/CAPA/변경관리에는 표준 용어가
있으며, "대충 비슷한" 답은 단지 덜 유용한 것에 그치지 않는다 —
컴플라이언스 결과를 낳는다. 이 문서는 1일차에 가지고 있었으면 했던
패턴 모음이다.

우리가 운영한 스택은 한국 제약사의 SOP 코퍼스와 구조화 QMS DB 위에서
돌아간다. 대부분의 내용은 문서 분류 체계가 있는 규제 산업(법무, 금융,
의료기기)에 일반화되지만, 예시는 제약이다.

## QMS 데이터 구조: semantic이 아니라 filter가 먼저

QMS는 문서 코퍼스가 아니다. **작고 고정된 분류 체계를 가진 테이블**이다:

- 일탈 (deviation)
- CAPA (corrective and preventive action)
- 변경관리 (change control)
- 시정조치
- 소비자불만

각 레코드는 구조화 필드를 갖는다: PR-ID, 등급(Major/Minor/Critical),
팀, 상태(진행/종결/반려), 작성일, 품목, 제목.

이 테이블에 대한 semantic 검색은 잘못된 기본값이다. "이번달 Major
일탈"을 묻는 사용자는 "Major 일탈"에 대한 nearest-neighbor 검색을
원하지 않는다 — `WHERE grade='Major' AND table='일탈' AND date_from=
<이번달>`을 원한다. QMS에 대한 semantic 검색은 **title 필드 한정**의
fallback이고, 그마저도 구조화 필터가 먼저 적용된 이후에야 쓰인다.

설계 원칙: 에이전트의 QMS 도구는 `table_filter`, `grade`, `team`,
`state`, `date_from/to`, `title_contains[]`, `exclude_terms[]`를
1급 파라미터로 받는다. 이 중 `title_contains[]`만 semantic이고 나머지는
exact 필터다. SOP 도구는 반대로 semantic이 먼저다 — 다른 분류 체계,
다른 도구.

## SOP 인용 규율: 세 필드 없으면 인용이 아니다

감사-추적 가능한 인용에는 세 가지가 모두 필요하다:

1. **문서 번호** — 예: `<COMPANY>-SOP-xxxx`. SOP 레지스트리의 기본키.
2. **버전** — rev. A/B/C 또는 시행일. 이게 없으면 "SOP 인용"은 의미
   없다. 조항 번호와 본문은 rev마다 바뀐다.
3. **조항** — 예: `5.1.4항`, `부록 2`. 감사자가 정확한 조항으로
   곧장 갈 수 있게.

셋 중 하나라도 빠지면 인용은 감사 불가능하다. 이 세 필드 요구를
생성 프롬프트, 참조 패널 스키마, 검증자 체크리스트에 모두 박아라.

나쁨: "SOP에 따르면..."
나쁨: "변경관리 절차에 명시되어 있다..."
좋음: "<COMPANY>-SOP-xxxx (Rev. B) 5.1.4항: '...'"

## 제형 fallback은 코드가 아니라 위키에 산다

한국 제약은 제품을 **제형**으로 분류한다: 고형제/주사제/액제/반고형제.
이 카테고리 레이블은 제품 제목에 **거의 리터럴로 등장하지 않는다**.
"A정 10mg"은 고형제지만 제목에 "고형제"라는 단어는 없다 — 제품명
토큰 "정"이 있을 뿐이다.

매핑:

- 고형제 → 정 / 캡슐 / 환 / 과립
- 주사제 → 주 / 주사 / 프리믹스
- 액제 → 시럽 / 액 / 점안
- 반고형제 → 연고 / 크림 / 젤

이 매핑을 검색 서비스에 **하드코딩하지 말라**. 이유는 둘:

1. 바뀐다. 신규 제품 라인이 새 토큰을 들여오고, 규제 분류는
   개정 간에 이동한다.
2. 하드코딩 매핑은 LLM 교체 후 조용히 썩는다 — 버그 패턴 5번
   ("하드코딩 라우팅") 참고.

에이전트가 literal 카테고리 검색에서 0건이 나왔을 때 `wiki_lookup`
액션으로 조회하는 위키 페이지에 두라. 올바른 플로우:

1. 사용자가 쓴 그대로 literal 카테고리 시도
   (`title_contains=["고형제"]`) — 사용자 표현 존중, 빠른 실패.
2. 0건이면 `wiki_lookup(query="제형 분류")` 호출.
3. 위키가 돌려준 토큰 집합으로 재시도.
4. 최종 thought에서 **치환 사실을 설명** — 검색이 무작위가 아니라
   의도적으로 넓혀졌음을 사용자에게 알린다.

이 방식은 도메인 지식을 편집 가능한 단일 위치에 두고, 에이전트가
자기가 한 일을 정직하게 말하게 하며, 모델 교체에 견고하다.

## 변경관리 체인: 상류 상태에 대해 추론해야 한다

규제 정보 한 조각은 여러 문서의 **체인** 안에 산다:

    제품표준서 → 마스터 제조 지시 및 기록서 → 실제 제조 배치 기록서

"이 제품 제조 공정 바뀌었나?"라는 쿼리는 현재 상태 문서에 대한
semantic 검색으로는 답할 수 없다. 에이전트는 상류에 대해 추론해야
한다: 현재 제품표준서 리비전은 무엇이고, 어떤 마스터 기록이 그것을
인용하며, 어떤 변경관리 PR이 그 변경을 승인했는가.

에이전트 설계상 이는 **문서 종류 인지(document-kind awareness)**를
갖춘 레시피 설계를 의미한다:

- 쿼리가 *변경*을 참조하면, 변경관리 테이블부터 검색하고 — 거기서
  인용한 제품표준서로 이동.
- 쿼리가 공정 질문이면, 제품표준서부터 검색하고 — 그걸 참조하는
  가장 최근 변경관리 레코드를 이어서 노출.

순진한 "전체 검색 후 rerank"는 변경 레코드를 제품표준서 청크에
묻어 놓치거나, 변경 티켓 홍수에 제품표준서를 놓친다. 체인은
**명시적으로 걸어야** 한다.

## QMS 서브셋 필터 패턴

우리가 배포한 가장 가치 있는 패턴. 문서 02와 03에서 깊게 다뤘고,
여기서는 제약 맥락으로 다시 정리한다 — 이 패턴은 **특히 QMS에서**
중요하다. 규제기관은 이렇게 묻는다: "원료 이물 관련 일탈 보여줘,
그 중 고형제만, 그 중 Q1에 종결된 것만." 모든 refine은 이전 결과의
서브셋이지 새 질문이 아니다.

세 부분:

1. **원본 행을 세션 상태에 유지** — 매 QMS 검색 후 상위 N개의
   `pr_id / grade / team / state / date / title`.
2. **다음 턴의 prev_context에 `[PRIOR QMS RECORDS — FILTERABLE]`
   블록으로 노출**.
3. **thought 안에서 필터링** — 에이전트 시스템 프롬프트 규칙:
   사용자가 서브셋을 요청하고 직전 레코드 블록이 있으면,
   `final_answer.thought`에서 그 행들을 필터링하고 **재검색하지
   말라**.

"고형제만"으로 재검색하면 "원료 이물" 토픽 필터를 잃고 5529건의
일반 레코드가 나온다 — 사용자가 보던 11건 대신. 이 단일 실패
모드를 막기 위해 전체 패턴이 존재하며, QMS에서는 사용자 분노
1위 원인이다.

## GMP 인지 답변 포맷: 거부/헤지/답변의 구분

규제 답변은 **세 가지 수용 가능한 형태**만 갖는다.

### 답변 (인용과 함께)

검색 결과가 질문을 직접 모호성 없이 커버한다. 문서 번호, 버전,
조항과 함께 관련 조항을 그대로 인용. 헤지 문구 없음. "일반적으로"
없음.

### 헤지 (부분 인용과 함께)

검색 결과가 질문을 **부분적으로만** 커버한다 — 예: SOP에 허용오차
범위는 기술되어 있으나 사용자가 물은 특정 구성품의 구체 수치는
없다. 명시적으로 말하라:

> <COMPANY>-SOP-xxxx 5.1.4항에 허용오차 기준이 기술되어 있으나, 사용자가
> 문의한 특정 구성품에 대한 구체 수치는 해당 조항에 명시되지
> 않았습니다. 관련 수치는 배치 기록서 또는 품질 기준서를
> 확인해야 합니다.

헤지는 **격차를 이름 붙이고 그 격차를 메울 수 있는 곳을 가리킨다**.

### 거부 (유용하게)

검색 결과가 질문을 전혀 커버하지 않는다. 거부는 "모릅니다"가
아니다. 거부는 사용자가 대신 참고해야 할 **가장 가까운 SOP를
가리킨다**:

> 해당 주제에 대한 직접 문서는 확인되지 않았습니다. 관련 가능성이
> 있는 문서는 <COMPANY>-SOP-xxxx (교정관리) 및 <COMPANY>-SOP-xxxx (측정장비
> 유지관리)이며, 구체적인 측정 절차가 필요하시면 해당 문서를
> 확인해 주십시오.

맨 "정보 없음" 거부는 컴플라이언스에는 안전하지만 사용자에게는
적대적이다. 포인터가 붙은 거부는 둘 다 만족한다.

경계해야 할 안티패턴: 실제로 관련 내용을 인용하고 있는 답변에
한국어 헤지 "직접적인 내용은 없으나, …"가 등장하는 경우. 이건
헤지가 아니라 — LLM이 부분 매칭 콘텐츠에 보수적 규칙을
과적용한 것. 프롬프트 수정법은 문서 05(한국어 NLP 함정) 참고.

## 시스템이 로깅해야 할 증거 추적(evidence trail)

규제 답변마다 시스템은 다음을 **재구성할 수 있어야** 한다:

- 에이전트가 호출한 도구 + 파라미터.
- 검색된 문서/레코드 ID + 점수.
- 생성 컨텍스트에 실제로 실린 서브셋.
- 답변의 어떤 주장이 어떤 증거로 추적되는가.
- 검증자 결과 (PASS/WARN + 사유).

세션 단위로 저장하고, 보존 기간은 최소한 규제기관의 감사 기간과
맞춰라. 감사자가 "그 날짜에 시스템이 사용자에게 왜 X라고 말했는지"
물었을 때, **로그만으로** 답할 수 있어야 한다.

## 요약

- QMS는 타입 있는 테이블이다. 필터 먼저, 검색은 나중에.
- SOP 인용은 문서번호 + 버전 + 조항. 셋 다.
- 제형 매핑 같은 표는 코드가 아니라 에이전트가 읽는 위키에 둔다.
- 문서 체인(제품표준서 → 제조 기록)은 명시적으로 걸어야 한다.
  flat rerank로는 복구되지 않는다.
- QMS의 서브셋-필터 후속 질문은 원본 행 유지 + thought에서 필터
  규칙이 없으면 11건 대신 5529건의 엉뚱한 PR을 돌려준다.
- 답변은 세 가지 형태 중 하나 — 답변 / 헤지 / 유용한 거부. 맨
  "정보 없음" 금지.
- 세션별 증거 추적을 전량 로깅하라. 감사는 반드시 묻는다.
